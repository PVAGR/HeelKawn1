## 1. WHAT HEELKAWN IS

HeelKawn is a deterministic Godot 4.6 colony sim / myth-engine. No randomness — all "random" behavior flows through WorldRNG seeded streams. The world records every event, derives meaning from consequences, and persists scars. Pawns have needs, skills, kinship bonds, life paths, and neural-net-weighted decision matrices. Settlements grow, collapse, revive, and leave ruins. The player can incarnate as any pawn or observe from above.

## 5. PERFORMANCE HOTSPOTS — WHAT WILL BREAK AT SCALE

### HOTSPOT 1: TickManager._call_tick_on_tickables() — EVERY TICK
**File**: autoloads/TickManager.gd:183-196
**Problem**: Every single tick:
1. `tree.get_nodes_in_group("tickable")` — full scene tree traversal
2. Builds new array `tickable_nodes`
3. Sorts by `str(a.get_path())` — string conversion + comparison
4. At 100x with 500 ticks/frame: 500 tree traversals + 500 sorts per frame

**Scale risk**: HIGH. With 100+ tickable nodes, this becomes the dominant per-tick cost.

### HOTSPOT 2: WorldMemory._events.remove_at(0) — O(n) shift
**File**: autoloads/WorldMemory.gd:297-300
**Problem**: When events hit the 50K cap, `_events.remove_at(0)` shifts all remaining elements. O(n) per dropped event.
**Scale risk**: MEDIUM. Only triggers at 50K events, but then every new event causes an O(n) shift.

### HOTSPOT 3: WorldMemory.to_save_dict() — deep copy of entire event array
**File**: autoloads/WorldMemory.gd:808
**Problem**: `_events.duplicate(true)` deep-copies the entire event array. Called by WorldMeaning.recompute() every 8-20 ticks.
**Scale risk**: HIGH. At 50K events, this allocates and copies 50K dictionaries every 8-20 ticks.

### HOTSPOT 4: WorldMeaning.recompute() — iterates all events
**File**: autoloads/WorldMeaning.gd (recompute method)
**Problem**: Called every 8-20 ticks. Iterates all events to build meaning_by_region. Uses to_save_dict() which deep-copies.
**Scale risk**: HIGH. Linear in event count, called frequently.

### HOTSPOT 5: SettlementMemory._living_pawns() — get_nodes_in_group every 30 ticks
**File**: autoloads/SettlementMemory.gd:1213-1224
**Problem**: `tree.get_nodes_in_group("pawns")` — full scene tree traversal. Returns empty array 29/30 ticks (throttled).
**Scale risk**: MEDIUM. Throttled but still traverses scene tree.

### HOTSPOT 6: Multiple get_nodes_in_group() callers
**File**: Various
**Problem**: BodyRiskManager (every 10 ticks), IntentMemory (every 60 ticks), ToolManager (every 200 ticks), KnowledgeSystem (every 300 ticks), WorldEventSystem (every 1000 ticks) all call get_nodes_in_group("pawns").
**Scale risk**: MEDIUM. Each is throttled, but they add up.

### HOTSPOT 7: Deep-copy getters in hot paths
**Files**: WorldMeaning.get_region_meaning(), SettlementMemory.get_settlements(), WorldPersistence.get_region_persistence()
**Problem**: `.duplicate(true)` on every read. SettlementMemory.get_settlements() is called by multiple systems.
**Scale risk**: MEDIUM. Unnecessary copies when callers only read.

### HOTSPOT 8: _recompute_first_tick_for_type — O(n) scan when event dropped
**File**: autoloads/WorldMemory.gd:380-382
**Problem**: When the first event of a type is dropped, scans ALL events to find the new first.
**Scale risk**: LOW. Only triggers on drop + type mismatch.

## 6. EVENT SCHEMA GAP — WHY MEANING TAGS DON'T EMERGE

### The Problem
WorldMeaning.recompute() requires events to have both `"r"` (region key int) and `"k"` (kind int) to process them. But FoodChainManager._check_famine_conditions() records events via the generic `WorldMemory.record_event()` with `"type": "famine_warning"` — no `"k"`, no `"r"`.

**Result**: 314 events exist at tick 2000, but WorldMeaning meaning_by_region is empty because none of the events pass the `e.has("k")` gate.

**The typed recorders** (record_starvation_event, record_pawn_death, etc.) DO include `"k"` and `"r"`, but they're rarely called in a healthy colony. The generic recorder is the dominant path.

### What This Means
- `hunger_place`, `repeated_death`, `safe_hearth` tags can never fire from famine_warning events
- The meaning pipeline is structurally disconnected from the dominant event source
- This is a design gap, not a bug

## 13. RECOMMENDED PERFORMANCE FIXES (priority order)

### Fix 1: Cache tickable nodes in TickManager (HIGHEST IMPACT)
**Problem**: get_nodes_in_group("tickable") + sort every tick. At 100x with 500 ticks/frame, this is 500 tree traversals + 500 sorts.
**Fix**: Maintain a cached list of tickable nodes. Update on tree_entered/tree_exited signals. Sort once, not every tick.
**Impact**: Eliminates the single worst per-tick allocation/computation pattern.

### Fix 2: Replace WorldMemory._events array with ring buffer
**Problem**: remove_at(0) is O(n). to_save_dict() deep-copies everything.
**Fix**: Use a ring buffer (circular array). Oldest events are overwritten, no shifting. For to_save_dict(), iterate the ring instead of deep-copying.
**Impact**: Eliminates O(n) shift at 50K cap. Reduces allocation in recompute path.

### Fix 3: WorldMeaning.recompute() — incremental, not full scan
**Problem**: Recomputes from scratch every 8-20 ticks by iterating all events.
**Fix**: Track dirty regions since last recompute. Only reprocess events added since last flush. Keep aggregate counters instead of scanning.
**Impact**: Reduces recompute from O(events) to O(new_events_since_last_flush).

### Fix 4: Fix event schema gap — connect famine_warning to meaning pipeline
**Problem**: FoodChainManager records famine_warning events without "k" or "r" keys. WorldMeaning ignores them.
**Fix**: Either make FoodChainManager use record_starvation_event(), or extend WorldMeaning.recompute() to also process events with "type" string keys.
**Impact**: Connects 314+ events to the meaning pipeline. Tags like hunger_place can actually fire.

### Fix 5: Eliminate deep-copy getters
**Problem**: get_region_meaning(), get_settlements(), etc. duplicate(true) on every read.
**Fix**: Return const references or read-only views. If mutation safety is needed, only duplicate when the caller actually modifies.
**Impact**: Reduces allocation in every system that reads settlement/meaning data.

### Fix 6: Batch get_nodes_in_group("pawns") calls
**Problem**: 6+ systems independently call get_nodes_in_group("pawns") at different intervals.
**Fix**: Maintain a cached pawn registry in PawnSpawner or a dedicated manager. Update on pawn spawn/death.
**Impact**: Eliminates redundant scene tree traversals.

## 15. THREE-LINE SUMMARY

HeelKawn is a ~50K-line deterministic Godot colony sim with 64 autoloads, 30+ game_tick listeners, and a working headless test suite. The simulation runs smooth at 100x in a young colony, but TickManager re-scans the scene tree every tick, WorldMemory deep-copies all events every 8-20 ticks, and the dominant event source (famine warnings) is disconnected from the meaning pipeline by a schema mismatch. The highest-impact fix is caching tickable nodes in TickManager; the most meaningful fix is connecting famine_warning events to WorldMeaning so regional tags can actually fire.
